Clinical Safety

What this software is, what it isn't, and the clinical safety documentation behind it. Please read before relying on any output.

⚠
Please read — the honest version

This software is built, maintained, and released by a single GP developer as a personal project, on a best-effort basis. I maintain a clinical safety case — an intended-purpose statement, a hazard log, and a clinical safety notice (all linked below) — and do my honest best to identify and mitigate risks. But this is not a commercially assured medical device and it comes with no warranty.

It is a supplementary aid that re-organises and surfaces information already in the Medicus record. It does not make clinical decisions, does not constitute clinical decision support, and is not a substitute for your own clinical judgement or for the live patient record. Every value it displays must be verified against the source record before you act on it.

Use it at your own discretion and risk. If you are not comfortable with these terms, please do not use it.

Accept clinical features for this practice

One practice-level acceptance that switches on the features that ship off by default: the reception guided-capture pathways, the Sentinel alert library (extra monitoring), and the Knowledge notice. Unlike the per-install attestations, this acceptance is carried in a suite backup, so restoring that backup on another machine propagates it across the practice.

Clinical safety case documents

The full documents live in the public repository and are regenerated weekly from the current codebase, so they always reflect the latest release.

Intended Purpose ↗ What the software is for, and what it is explicitly not for. Clinical Safety Notice ↗ Required reading: conditions of safe use, known limitations, and the single most important rule. Hazard Log ↗ Identified hazards, their controls/mitigations, and residual risk assessment. Full Disclaimer & Terms ↗ The binding terms of use. In any conflict, the disclaimer prevails.

CQC Inspection Readiness Safe & Well-led evidence

Turns the suite's own monitoring rule-set and its dated currency into evidence for the CQC Safe and Well-led key questions — built from the rules the suite ships, with no patient data. Two modes: an internal Readiness check (“are we ready?” with what to fix), and a sign-off-stamped Evidence export for your wider pack, produced only after you confirm you have reviewed the figures. It is supporting evidence, not proof of compliance, and does not replace the inspection or the people-experience / staff-feedback parts of a pack.

It opens in its own tab so it prints cleanly to PDF and exports to CSV. (Also reachable any time from the command palette — press Ctrl+K and type “CQC”.)

Event ledger machine-local

A record of what this extension displayed or did on this computer — alerts shown, dismissals, recall tasks created, patient summaries copied, pre-flight checks run, and results filed via Lab Filing. It exists to answer “did the tool flag this?” with evidence.

What this is: a machine-local log of what this extension showed and did while it was open on this machine only. Patients appear as their Medicus UUID — never by name.

What it is not: it is not part of the clinical record and is no substitute for Medicus’s own audit trail. Absence of an event is not evidence that nothing was shown — the extension only logs while it is open and running on this machine.

Retention: capped at 5,000 events and 90 days — older events are pruned automatically. It never leaves this machine and is deliberately excluded from suite backups (the same rule as the Lab Filing audit log): restoring it onto another machine would fabricate a misleading record of what was shown there.
Deletes every event on this machine. Cannot be undone.

Suite health self-diagnosis

The suite depends on specific pieces of Medicus's own page structure to work — a checkbox, a button, a row of results. This section is the extension quietly checking that those pieces are still where it expects, using a small background probe that runs on the live Medicus page. That probe counts elements only — it never reads patient data or page text. A degraded row means Medicus's interface may have changed since the extension was built, and that one feature may be silently limited until the extension is updated to match. It is not a fault report about Medicus — Medicus changing its own UI is normal and expected; this is the suite catching up with it, automatically, before a clinician has to notice the hard way.

What to do about a degraded row: first check you're on the latest version of the extension (see the version badge above, or the update checker) — most fixes for this ship as a routine update. If it's still degraded after updating, tell Dave (the extension's maintainer) which feature it is so the underlying selector can be fixed.

Some contracts below are marked fixture-tested only rather than live-checked — their own selectors are too generic, or too transient (existing only for a few seconds mid-interaction), to probe live without false alarms. Those are still regression-tested against recorded Medicus DOM snapshots on every code change; they are just not re-checked against the live page every session. The explanation column says why, contract by contract.

▶ How to set up shared-folder deployment
1. One-time: put the extension on your shared drive
  1. Copy the entire Medicus Suite extension folder to a shared network location that every PC at the practice can open — for example S:\Medicus Suite\extension.
  2. The practice manager needs write access to that folder. Everyone else only needs to be able to read from it.
  3. Make sure the folder contains manifest.json at its top level — that is the file Edge and Chrome use to recognise an extension.
2. Add the extension to each PC (about 2 minutes, once per PC)
  1. Edge (NHS default): Open Edge and type edge://extensions in the address bar, then press Enter. In the top-right corner, turn on the Developer mode toggle. Click Load unpacked. In the window that opens, navigate to the shared extension folder and click Select Folder.
  2. Chrome: The steps are identical — type chrome://extensions instead, then follow the same Developer mode → Load unpacked steps.
  3. You should now see the Medicus Suite icon in the browser toolbar. Click it (or open the side panel) to confirm the extension is running.
  4. Repeat on every PC that needs the extension. Each PC points to the same shared folder.
3. Practice manager: change settings for everyone (about 1 minute)
  1. On your own PC, set up each module the way the practice wants it — practice code, monitoring rules, Knowledge tab entries, triage keywords, whatever you need.
  2. Come back to this Settings page, scroll up to Publish settings to the practice, fill in a label (e.g. "Acme Surgery — June 2026") and your name or email, tick the modules you want to share, and choose how each one should apply:
    • Fill gaps only — adds your settings on PCs that haven't customised them yet; never changes what someone has already set themselves.
    • Enforce for everyone — everyone gets exactly your version, replacing their local changes for that module. Use this when you need to push a correction that must take effect for all staff.
  3. Click Publish to shared folder. The first time, a file-save window opens — navigate to the shared extension folder and save the file there (replace the existing practice-profile.json if asked). After the first time, the extension remembers the location and publishing is a single click.
4. What happens on everyone else's PC
  1. There is nothing for other staff to do. Each PC checks the shared folder roughly every 15 minutes while the browser is open, and also checks on every browser start.
  2. When it finds a new version, it quietly applies the updated settings in the background. Staff do not need to restart their browser or do anything at all.
  3. If the browser is closed, the changes will be picked up the next time it starts.
  4. New versions of the extension itself (when you update the files in the shared folder) install themselves automatically when the PC is next idle — no manual reinstall needed.
5. If something isn't working
  • Is the PC running the extension from the shared folder? Open edge://extensions, find Medicus Suite, click Details, and look under Source or Loaded from. It should show the shared folder path, not a local path on that PC.
  • Is practice-profile.json in the right place? The file must sit in the same folder as manifest.json — not inside a subfolder. You can check by opening the shared extension folder and looking for both files side by side (the practice-profile.json file is a JSON text file).
  • Test it immediately: Press Check for update on this page. If an update is found it will say so; if not, the file is either missing or the version hasn't changed.
  • A colleague's setting didn't change: The module is probably set to Fill gaps only and they had already customised that setting. Go back to Publish settings to the practice, change that module to Enforce for everyone, and publish again.
  • Track what each PC has applied: The status box at the top of this card shows exactly which version this PC has applied, and when.
