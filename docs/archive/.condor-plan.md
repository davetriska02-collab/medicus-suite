# Condor Tab — Implementation Plan

## What We're Building

A new **Condor** nav tab — a practice operational intelligence layer that aggregates
multiple live data streams into a single high-density view. It surfaces genuinely novel
cross-stream insights (Practice Pressure Index, Demand-Capacity Gap, submission velocity,
task age distribution, clinician workload balance, referral rate) that no existing module
provides.

## Architecture

```
side-panel/modules/condor/
  condor.js          — init/cleanup/poll, data fetch, layout assembly, imports all card renderers
  condor-data.js     — all API fetching: slots, WR, submissions, request-monitor, activity, referrals
  condor.css         — complete styles for all cards (shared layout + per-card)
  cards/
    ppi.js           — Practice Pressure Index SVG arc gauge
    demand-gap.js    — Demand-Capacity Gap ratio card
    velocity.js      — Submission Velocity 24-hr histogram
    task-age.js      — Task Age Distribution bucketed bar
    workload.js      — Clinician Workload Balance chart
    referral-rate.js — Referral Rate per Clinician
    waiting-room.js  — Waiting Room summary card
    day-score.js     — Day Score + 30-day sparkline (with condor-io.js persistence)
    activity.js      — Today's Activity footer strip
```

### Data contract — `data` object passed to all card render functions

```js
{
  siteId: string | null,
  slots: {
    entries: [{ staff: string, type: string, startDateTime: string }],
    amRemaining: number,   // slots with hour < 12 and not past
    pmRemaining: number,
    totalRemaining: number,
    byStaff: { [staffName]: { amRemaining: number, pmRemaining: number } }
  } | null,
  waitingRoom: {
    appointments: [{ patientName: string, start: string, reason: string, deliveryMode: string, isArrived: boolean }],
    arrivedCount: number,
  } | null,
  submissions: {
    tasks: [{ id: string, type: string, createdAt: string, hourOfDay: number }],
    // type: 'medical'|'admin'|'rxRoutine'|'rxNonRoutine'|'investigation'
    totals: { medical: number, admin: number, rxRoutine: number, rxNonRoutine: number, investigation: number, all: number },
    byHour: [{ hour: number, medical: number, admin: number, rxRoutine: number, rxNonRoutine: number, investigation: number }],  // 24 entries
  } | null,
  requestMonitor: {
    items: [{ id: string, patient: string, summary: string, priority: string, createdAt: string, ageMs: number }],
    urgentCount: number,
    totalCount: number,
    byAgeBucket: { lt1h: number, h1to4: number, h4to8: number, gt8h: number }
  } | null,
  activity: {
    rows: [{ name: string, consultations: number, routineRx: number, nonRoutineRx: number, reviews: number, documents: number, results: number, total: number }],
    totals: { consultations: number, routineRx: number, nonRoutineRx: number, reviews: number, documents: number, results: number, all: number }
  } | null,
  referrals: {
    available: boolean,
    rows: [{ clinician: string, count: number, consultations: number, rate: number }],
    practiceAvgRate: number
  } | null,
  capacityPreset: { minimum: number, status: string } | null,
  fetchErrors: string[]
}
```

### Card file contract

Each card file in `cards/` must:
- Export exactly one named function: `export function renderXxx(data) { return '<html string>'; }`
- Accept the full `data` object defined above (use only the fields it needs)
- Return an HTML string (no DOM manipulation — purely string-based)
- Handle null gracefully: if its required stream is null, return an appropriate placeholder card

### condor.js orchestration contract

condor.js imports and calls all card renderers. Shell layout (which each unit must respect):

```html
<div class="condor-wrap">
  <div class="condor-hero">       <!-- PPI gauge — full width --></div>
  <div class="condor-grid">
    <div class="condor-col">      <!-- left: WR + slots supply --></div>
    <div class="condor-col condor-col-wide"> <!-- centre: velocity + gap + task age --></div>
    <div class="condor-col">      <!-- right: workload + referral rate --></div>
  </div>
  <div class="condor-footer">     <!-- day score + activity strip --></div>
</div>
```

### Unit 2 — PPI Gauge Card
**File:** `side-panel/modules/condor/cards/ppi.js`
**Change:** REPLACES the stub. Implements the Practice Pressure Index: weighted sum of (a) arrivedCount
from WR (30%), (b) submissions.totals.medical + admin (25%), (c) requestMonitor.urgentCount (25%),
(d) inverse of slots.totalRemaining vs capacityPreset.minimum (20%). Scores 0–100. SVG arc gauge
(same approach as trend-chart.js — build the SVG path as a string, no library). Green <40, amber 40–70,
red >70. Shows breakdown chips below the gauge. Exports `renderPpi(data)`.

### Unit 3 — Demand-Capacity Gap Card
**File:** `side-panel/modules/condor/cards/demand-gap.js`
**Change:** REPLACES the stub. Computes open medical request count (submissions.totals.medical) vs
slots.totalRemaining. Ratio displayed as a large number with two sub-lines. Green <1.0, amber 1.0–1.5,
red >1.5. Shows the raw figures (e.g. "31 requests / 17 slots"). Exports `renderDemandGap(data)`.

### Unit 4 — Submission Velocity Card
**File:** `side-panel/modules/condor/cards/velocity.js`
**Change:** REPLACES the stub. Renders a 24-bar stacked SVG histogram (one bar per hour, stacked by
task type: medical/admin/rxRoutine/rxNonRoutine/investigation). Uses submissions.byHour[]. Colors match
existing chip palette. X-axis shows hour labels (06, 09, 12, 15, 18). Exports `renderVelocity(data)`.

### Unit 5 — Task Age Distribution Card
**File:** `side-panel/modules/condor/cards/task-age.js`
**Change:** REPLACES the stub. Takes requestMonitor.byAgeBucket (lt1h, h1to4, h4to8, gt8h) and renders
a horizontal stacked bar showing proportion in each bucket. Colors: green for <1h, amber for 1-4h,
deep amber for 4-8h, red for >8h. Shows count labels. If any items are in gt8h bucket, adds a red
warning line with count. Exports `renderTaskAge(data)`.

### Unit 6 — Clinician Workload Balance Card
**File:** `side-panel/modules/condor/cards/workload.js`
**Change:** REPLACES the stub. Takes activity.rows and slots.byStaff. For each clinician: renders a
horizontal bar with actual output today (consultations + Rx + reviews) stacked in segments, and
appends remaining slots as a lighter "ghost" bar. Sorts by total output descending. Uses `esc()` from
trend-chart.js for all text. Exports `renderWorkload(data)`.

### Unit 7 — Referral Rate Card
**File:** `side-panel/modules/condor/cards/referral-rate.js`
**Change:** REPLACES the stub. If referrals.available is false, renders a single-line "Referrals data
unavailable — visit the Referrals Tracker page first" message. Otherwise renders a horizontal bar chart:
one bar per clinician, bar length = referral rate %, practice average shown as a dashed vertical line.
Clinicians >50% above practice average get an amber warning marker. Exports `renderReferralRate(data)`.

### Unit 8 — Waiting Room Card
**File:** `side-panel/modules/condor/cards/waiting-room.js`
**Change:** REPLACES the stub. Renders WR summary: large arrived count, list of arrived patients (name
anonymised to first name + last initial, delivery mode icon, wait time from start to now). Shows
pending (booked but not arrived) count below. Exports `renderWaitingRoom(data)`.

### Unit 9 — Day Score + Sparkline Card
**File:** `side-panel/modules/condor/cards/day-score.js`
**Change:** REPLACES the stub. After 17:00 computes a 0–4 day score (each criterion: submissions stayed
below red threshold? urgent tasks all <8h old? remaining slots > 0? referral rate within 1.5x avg?).
Reads/writes `condor.dayScores` array (last 30 entries) from chrome.storage.local. Renders the score
as 4 tick/cross icons + a 30-day SVG sparkline (one bar per day, colored by score). Exports
`renderDayScore(data)`.

### Unit 10 — Activity Footer Card

## E2E Test Recipe

This is a Chrome MV3 extension — headless browser testing is not available in the CI environment.

**For all units:**
1. Run `node test-*.js` from the repo root. All existing tests must remain green.
2. Unit 11 must additionally pass the new condor-io test it adds to test-suite-io.js.
3. For UI card units (2–10): code-review skill is the verification pass (no browser available).
   Workers should verify: correct exports match the contract, all template literals use `esc()` for
   any user-derived strings, null data cases return a valid HTML placeholder, no `document.*` or
   `chrome.*` calls (card files are pure rendering functions).

**e2e is skipped for UI cards** — Chrome extension loading requires a real browser which is not
available to workers. The code-review skill plus the above structural checks are the gate.

## Codebase Conventions (all workers must follow)

1. **Copyright header:** `// © 2026 Graysbrook Ltd. Proprietary — all rights reserved. See LICENSE.`
2. **`'use strict';`** at the top of every JS file
3. **No comments** unless the WHY is non-obvious. No docstrings.
4. **XSS safety:** All user-derived / API-derived strings rendered into HTML must go through `esc()` imported from `../shared/trend-chart.js` (or `../../shared/trend-chart.js` from cards/ subdirectory). Numeric values that are provably numbers (e.g. `Number(x).toFixed(1)`) are safe without esc.
5. **CSS variables:** Use only `var(--accent)`, `var(--red)`, `var(--amber)`, `var(--green)`, `var(--border)`, `var(--text-3)`, `var(--t4)`, `var(--t1)`, `var(--sans)` — these are defined in panel.css.
6. **No external libraries.** SVG charts built as template-literal strings, same approach as `shared/trend-chart.js`.
7. **No `document.*` or `chrome.*` in card files** — cards are pure `(data) => htmlString` functions.
8. **condor.js module pattern:** Export `async function init(el)` returning cleanup function, and `export function cleanup()` — same pattern as all other modules.
9. **Node-compatible IO files:** condor-io.js must end with `if (typeof module !== 'undefined' && module.exports) { module.exports = { condorExport, condorImport }; }` (same as submissions-io.js).
10. **Git branch:** All workers develop on branch `claude/modest-carson-bEdZG` and push there.
