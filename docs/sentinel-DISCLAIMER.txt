 1. NATURE OF THE SOFTWARE
--------------------------------------------------------------------------------

Sentinel is a passive, read-only memory aid that re-displays, in a sidebar
inside the user's own browser, values already present in the Medicus
electronic patient record for the patient the user is already viewing.

Sentinel:

  - performs no clinical reasoning;
  - produces no clinical recommendation;
  - performs no triage, prioritisation, ranking or scoring of patients;
  - performs no calculation that changes the meaning of any clinical value;
  - performs no action on any patient record or any external system;
  - writes nothing, sends nothing, submits nothing, modifies nothing;
  - transmits no data outside the user's local browser;
  - stores no patient data persistently;
  - operates only while the user is authenticated to Medicus in their own
    browser session.

The frozen intended-purpose statement is reproduced at section 3 below. Any
feature or use case beyond that statement is outside the scope of Sentinel and
the disclaimers in this document do not apply to such use.

--------------------------------------------------------------------------------

The author asserts, based on a good-faith reading of the MHRA guidance on
software as a medical device current at the date of this document, that
Sentinel falls outside the scope of the UK medical devices regulations. This
assertion rests on the following facts:

 3. INTENDED PURPOSE (FROZEN STATEMENT)
--------------------------------------------------------------------------------

  Software that displays, against the patient's active medication list, active
  problem list, and recent observations as already recorded in the Medicus
  electronic patient record, the most recent recorded values relevant to
  published drug-monitoring guidance and to QOF 2025/26 indicator criteria,
  and indicates whether those values fall within the recommended interval or
  whether the relevant QOF indicator is achieved.

  The software does not recommend clinical actions, does not order
  investigations, does not write to the patient record, does not modify QOF
  claims data, does not transmit any data outside the user's browser, does
  not analyse images, does not generate synthetic data, and does not
  constitute clinical decision support.

  It is a passive display tool for use by the clinician as a memory aid. All
  clinical decisions, including verification of any displayed value against
  the source record, remain the responsibility of the clinician.

Any use of Sentinel for any purpose outside the above statement is at the
user's sole risk and is not covered by the protections of this disclaimer.

 6. KNOWN LIMITATIONS
--------------------------------------------------------------------------------

The user acknowledges that Sentinel has the following known limitations,
each of which may cause Sentinel to display information that is incomplete,
inaccurate, out of date or misleading:

  (a) QOF register membership is determined by substring matching on
      problem labels, not by SNOMED refset membership. False positives and
      false negatives are inherent to substring matching.

  (b) QOF indicator coverage is a curated subset of the full QOF 2025/26
      specification. Indicators not in the curated set will never appear.
      Indicators in the set may not reflect every nuance of the official
      business rules including exception coding.

  (c) The investigation dashboard endpoint does not contain every coded
      entry in the patient record. Encounter-coded entries (e.g. annual
      review codes, smoking status, mental health questionnaires) may not
      appear and indicators relying on them may report "no data".

  (d) Annual QOF specification changes are not automatically detected.
      The user must review the rule files at each contract refresh.

  (e) User edits to thresholds and intervals override the defaults and
      may cause Sentinel to display different statuses than the published
      guidance would indicate.

  (f) Medicus may change the shape of its API endpoints at any time
      without notice, in which case Sentinel may display incomplete or no
      data until the normalisers are updated.

  (g) Sentinel does not validate the correctness of the Medicus record.
      If the source data is wrong, the displayed evaluation will be wrong.

  (h) Sentinel only operates within the Medicus EHR. It produces no
      useful output on other EHRs or on any non-Medicus web page.

The user must read the README.md document in full before installing and
using Sentinel, and must accept these limitations as conditions of use.

--- v1.1.0 addendum ---

Custom drug-monitoring rules authored through the built-in form builder remain within the stated intended purpose of this software. Custom rules use the same passive threshold-based display engine as the bundled rules: they re-display values already present in the patient record, indicate whether those values fall within the user-configured interval, and produce no clinical recommendation. The form builder does not expose arbitrary clinical decision logic. All constraints documented in the intended-purpose statement above apply without modification to custom-authored rules.

Backup and restore functionality moves the user's own configuration data within the user's own browser storage. No patient data is included in any backup file. No data leaves the user's browser except by the user's own deliberate file-download action.

The migration of the Triage Lens storage key from config to triagelens.config is an internal namespace change with no effect on clinical function or data handling.

As with the bundled rule set, user-authored custom rules may become out of date as clinical guidance changes. The user retains full responsibility for reviewing custom rules at appropriate intervals and for verifying every displayed value against the source patient record before taking any clinical action.


--- v1.2.0 addendum ---

Custom clinical indicators authored through the form builder remain within the stated intended purpose of this software. They are passive threshold checks against values already recorded in Medicus. They produce no clinical recommendation, no record write, and no data egress. The form constrains the user to three engine-supported check shapes (observation-threshold, medication-present, observation-recent) and to the bundled QOF register set.

Custom indicators are not QOF indicators. They do not constitute QOF achievement and must not be relied upon for QOF income claims. They are labelled visually as Custom and their points field (if set) is metadata only; it does not contribute to any QOF total in the UI. User-authored thresholds may diverge from any published guideline. The clinician retains responsibility for verifying the threshold and for verifying every displayed value against the source patient record before taking any clinical action.

The hotfix in v1.2.0 removes a developer-side hardcoded practice code that was preventing the extension from working correctly on practices other than the developer's own. The fix is data hygiene only. It does not alter the intended purpose, the clinical behaviour, or any data-handling characteristic of the software. The user's practice code, configured through the Options page, is the sole source for API endpoint selection. No practice code is preconfigured.


--- v3.77.5 addendum ---

This addendum records the state of the software at Medicus Suite v3.77.5. The
following features and changes have been added since the v3.26.4 addendum. In
each case the feature remains within the intended purpose at section 3 and does
not alter the fundamental nature of the software, its regulatory status, or its
data-handling characteristics. All obligations stated in this disclaimer and in
prior addenda continue to apply in full.

Pre-clinic Monitoring Sweep and Action Packs (v3.37.0–v3.56.0). The Sweep
module runs the existing Sentinel rules engine against today's booked patients
before clinic, producing a priority-ordered worklist of patients with
action-needed monitoring, QOF, or vaccine chips. The Sweep is a morning-huddle
aid only and shares all the limitations of the Sentinel monitoring module: the
"no chip does not mean monitoring is complete" principle applies with full force
to the Sweep output. From v3.44.0 it generates a printable reception handout of
copy-ready booking instructions per patient; from v3.48.0 the Sentinel panel
gains Action Packs (copy-ready blood-form requests, recall SMS messages, letters
and tasks per chip); from v3.56.0 the Sweep generates batch Action-Pack artefacts
for the full morning worklist. Every batch artefact carries the patient name and
NHS number and must be verified against the destination Medicus record before use:
wrong-patient paste risk is higher in batch than in single-patient use. None of
these artefacts are automatically submitted to any system; they are copy-ready
text only. Sweep results persist as a local working copy for up to two hours to
allow resume after an accidental tab switch; a resumed sweep reflects records at
the time the sweep ran and must not be treated as current.

Reception guided capture (v3.38.0). The Reception module presents a fixed
question set to non-clinical front-desk staff for a set of presenting problems,
with red-flag questions asked first. It produces a plain-text complaint
description for manual pasting into Medicus. It is not a triage tool: red-flag
prompts are escalation triggers only, not exclusion criteria, and a full set of
"no" answers does not mean the contact is routine. The receiving clinician reviews
every capture. All capture pathways ship disabled and require explicit practice
administrator enablement after clinician review. The user must confirm the correct
patient record before pasting generated text.

Condor operational dashboard (v3.35.0) and the Today tab (v3.60.0). Condor
displays administrative and operational practice metrics — waiting-room arrivals,
demand/capacity ratios, submission velocity, Practice Pressure Index, task-inbox
age — as an operational situational-awareness tool. The Today tab aggregates the
same administrative feeds into a single morning glance view. Neither tool makes
clinical decisions, assesses individual patients, or produces clinical output.
The Practice Pressure Index is an arithmetic composite of operational indicators
and does not assess clinical urgency or patient safety.

Triage Lens red-flag engine expansion (v3.46.0+). From v3.46.0 the free-text
triage-lens rule set was substantially expanded across clinical domains (stroke/TIA,
sepsis, anaphylaxis, meningitis, pregnancy emergencies, paediatric presentations,
2WW cancer flags, mental health, and others). These are supplementary pattern-
matching prompts surfaced for non-clinical staff handling incoming patient requests;
they are an aide-memoire, not a triage system. Absence of a chip is not absence of
risk; every contact must be reviewed by a clinician. The rule set cannot be
exhaustive. From v3.52.0 the engine was hardened with dropped-pattern warnings and
curly-quote normalisation; neither change alters the software's fundamental
character or user obligations.

Patient Record Visualiser SMR tab (v3.51.0). The SMR tab adds an anticholinergic
cognitive burden (ACB) score using the Boustani scale and a subset of STOPP/START
v3 prescribing flags. The ACB score is an arithmetic sum of per-drug published
burden scores; it is not a validated clinical assessment instrument. The
STOPP/START criteria implemented are a starter subset; absence of a flag does not
guarantee prescribing safety. Both outputs are supplementary aids to the clinician's
own prescribing assessment and to Medicus's own prescribing-safety systems.

Investigation Results queue triage chips (v3.62.0–v3.64.0). From v3.62.0 the
Triage Lens decorates rows in the Medicus Investigation Results filing queue with
per-row severity chips (urgent, abnormal) derived solely from the laboratory's own
reference-range and urgent-review flags, an under-prioritised flag, and an
unmatched-patient flag. There is deliberately no "normal" or "safe to file" chip:
absence of a chip asserts nothing about result normality. From v3.63.0 users may
author analyte-threshold rules that escalate a result's severity (for example
"Potassium >= 6.0 -> red"); such rules arrive disabled and require clinician review
before firing; they can only raise severity, never lower it or suppress the
laboratory's own flags. From v3.64.0 a built-in text-classification rule flags
MSU/urine-culture results as "Needs review" (amber) or shows a "No growth" info
chip based on configurable normal-phrase matching; the "No growth" chip is an
information chip — it is not a verdict that the result is safe to file. Users must
be alert to sterile pyuria and to laboratory comments recommending repeat culture
even when culture is negative. Every result must be opened and reviewed in Medicus
regardless of chip status; the chips only assist with prioritisation of that review.

Navigation API detection hardening (v3.74.1). The patient-change detection
mechanism used by the Investigation Results queue and the Triage Lens was hardened
by adding the browser Navigation API alongside the existing MutationObserver, with
a backstop polling interval, to reduce the risk of a missed patient-switch event.
This is a safety-engineering improvement to the wrong-patient guard; it does not
alter any user obligation.

Built-in result-rule expansion (v3.75.0–v3.77.2). A bowel-cancer-screening
non-responder text rule (v3.75.0) uses a flag-if-present "abnormalText" primitive:
it fires if the result text does not contain a configured normal phrase. By design,
this primitive cannot suppress a positive screening result. Six further built-in
threshold rules were added disabled by default (v3.76.0): lithium-level toxicity,
digoxin-level toxicity, critical low potassium, high adjusted calcium
(hypercalcaemia), low eGFR CKD G4 band, and a blood-culture calmer. These rules
ship disabled and must be explicitly enabled by a clinician who has reviewed their
clinical appropriateness for the practice's case mix; they are not activated by any
user configuration change other than deliberate enablement. Four further Keeper
rules were added disabled (v3.77.0) then enabled after CSO sign-off (v3.77.2):
hypocalcaemia, hypomagnesaemia, high TSH, and suppressed TSH. The TSH rules carry
a "suppressIfProblem" gate that suppresses the chip for patients with a coded
thyroid diagnosis. Users must understand that this gate is a heuristic that does
not replace clinical assessment: it may suppress a chip for a patient with a
thyroid diagnosis who still requires TSH monitoring. The clinician is responsible
for verifying that suppression is appropriate for each individual patient. All
result rules are escalate-only and cannot lower or suppress the laboratory's own
flags. Absence of a chip still asserts nothing about result normality.

Tombstoning and import hardening (v3.77.3–v3.77.5). The "removedBuiltins"
tombstone mechanism (v3.77.3) ensures that a built-in result rule deliberately
deleted by the user is not silently resurrected when the shipped configuration is
updated. Import hardening (v3.77.5) extended the 10 MB backup-import size cap and
per-field type validation to all per-module import paths, reducing the risk of
corrupt or maliciously-crafted import data reaching the rules engine. These are
data-integrity controls; they do not alter the intended purpose or any user
obligation.

User obligations specific to features in this addendum:

(a) The user must treat the Sweep worklist and Action-Pack artefacts as
    supplementary aids to clinical workflow, not as an authoritative record. Every
    artefact must be verified against the destination patient record before use.

(b) The user must ensure that Reception guided-capture pathways have been reviewed
    by a clinician before enablement, and must confirm the destination patient
    record before pasting generated text.

(c) The user must open and review every result in Medicus independently of any
    chip shown by the Investigation Results queue decoration. Absence of a chip
    does not indicate that a result is normal, safe to file, or has been reviewed.

(d) The user must not enable the v3.76.0 disabled-by-default threshold rules
    without reviewing their clinical appropriateness for the practice's case mix.

(e) Where TSH chips are suppressed by the suppressIfProblem thyroid-diagnosis gate
    (v3.77.2), the user must satisfy themselves independently that TSH monitoring
    is not required for that patient, and must not rely on the absence of a chip
    as evidence that monitoring is not needed.


--- v3.26.4 addendum ---

This addendum records the state of the software at Medicus Suite v3.26.4. The
following features have been added or expanded since the v3.4.1 addendum. In
each case the feature remains within the intended purpose at section 3 and does
not alter the nature of the software, its regulatory status, or its data-handling
characteristics.

BP Trend and ACR Trend (v3.25.0) are new side-panel display modules that render
graphical trend charts of blood pressure and albumin-creatinine ratio (ACR)
history, drawing exclusively on observation data already fetched by the Monitoring
module from the Medicus investigation dashboard. The ACR Trend module displays
threshold-based action banners (e.g. "ACR ≥70 mg/mmol — consider nephrology
referral per NICE NG203") that are arithmetic comparisons against published
guidance, worded as prompts to consider and verify, not clinical decisions made by
the software. No new external endpoints are called; no data is transmitted
externally.

Falling eGFR trend and hyperkalaemia (K⁺) alerts (v3.18.0) and a rising HbA1c
trend monitor (v3.19.0) use the existing observation-trend engine and a new
observation-alert check kind to surface threshold-based and trend-based prompts
against observations already fetched from Medicus. These follow the same passive
display model as existing drug-monitoring and QOF rules: they re-display a value
or trend already in the patient record against a published threshold (NICE NG203,
UK Kidney Association guidance, NICE NG28/NG17); they make no clinical
recommendation and produce no output that constitutes a diagnosis.

ADHD medication monitoring rules, smoking status indicators, and carbamazepine
monitoring (v3.26.x) follow the same passive drug-monitoring model as existing
bundled rules. They use the existing engine without modification to intended
purpose.

Flu and COVID-19 vaccination eligibility alerts (v3.26.0) infer eligibility and
vaccination status (DUE / GIVEN / DECLINED) from coded problems, observations,
medications, QOF register membership, and BMI already present in the patient's
Medicus record, using JCVI/UKHSA 2025/26 published criteria. The inferred status
is not authoritative: vaccination administered outside the practice may not be
recorded locally. The user must verify vaccination status against the Medicus
record and, where applicable, national vaccination records before taking any
clinical action. Every chip carries a "DOUBLE-CHECK ELIGIBILITY" notice. This
feature does not connect to the National Immunisation Management Service (NIMS)
or any external data source; no data is transmitted externally.

Per-rule hide/snooze (v3.26.3) allows a user to permanently suppress individual
drug-monitoring or QOF indicator chips on their workstation, or to snooze a
vaccine chip until the season end. Suppression stores only rule identifiers in
chrome.storage.local, which contains no patient data. The user must understand
that a suppressed chip does not indicate that the monitoring has been completed:
the obligation at section 5 to verify every displayed value against the source
record applies with equal force to chips that are suppressed as to chips that
are active. Hiding a chip is not a substitute for completing the monitoring.

The Custom Alert Builder (v3.22.0) now exposes full parity with the rules engine,
adding the observation-alert check kind and additional filter fields
(requiresProblem, requiresAnyProblem, excludeIfProblem, sex, age range,
mustNotBePresent, per-test SNOMED aliases). These fields expose no capability
beyond the existing intended purpose; all obligations in the v1.1.0 and v1.2.0
addenda apply without modification to rules authored using these fields.

Journal-coded observations (v3.21.0) are now included in the side-panel
evaluation path. This extends the data source consulted by the Monitoring module
to include consultation and journal-coded entries already present in Medicus;
it does not consume any new API endpoint or transmit any data externally.
